# Feature — Read-only Mapper (авто-карта с трекингом позиции)

## Context

У клиента сейчас **нет никакого понятия о комнате/карте/позиции игрока** (grep по `go/` и `ui/`
даёт только timezone-код; GMCP/MSDP не поддержаны — транспорт `go/internal/session/telnet.go`
знает только Telnet + MCCP2). Движение — это пользовательские хоткеи/алиасы, шлющие команды
направлений.

У пользователей есть **приватные карты мира** в формате стороннего делфийского маппера
**MUD Mapper (mm25x)** — по одному `.mm2`-файлу на зону, обычно упакованные в архив. Задача:
дать залить такой архив в клиент, получить **read-only авто-карту**, которая следует за игроком
без GMCP (авто-ресинк по фингерпринту имени/описания + dead-reckoning по шагам), и уметь держать
**несколько наборов карт** (для разных миров), выбирая нужный в сессии.

Формат `.mm2` полностью реверс-инжинирнут — см. **Reference artifacts** ниже и раздел
**Факты о формате `.mm2`**. Приватные карты и артефакты в репозиторий не коммитятся.

### Reference artifacts (передаются отдельным бандлом, НЕ в этом репозитории)
Каталог `mapper-artifacts/` доставляется реализаторам отдельно:
- `FINDINGS.md` — полный разбор формата `.mm2` / модели `TMudRoom2`.
- `rmud_locate.py` — эталонный Python-парсер DFM-стрима + логика локализации (fingerprint-матчинг)
  и grid/world-BFS. Первоисточник для порта в Go (парсер + резолвер + neighbor-поиск).
- `webmap/` — прототип canvas-рендерера карты (vanilla JS, ноль зависимостей): зоны, зум/пан,
  этажи, швы, эмодзи-спецкомнаты, DT, поиск, отрисовка тамбнейлов комнат. Первоисточник для
  порта в TS-модуль.
- `rmud_index.json` + `krynn_slim.json` — пример экспортированного корпуса (эталон для golden-теста
  парсера) и slim-формат обмена (ключи `z t h x y l e d ch a p i s` + `dx dy dl`).
- `thumbs/` — пример картинок комнат (полные PNG + тамбнейлы) из внешнего LLM-пайплайна — эталон
  для схемы `room_images`.

## Факты о формате `.mm2` (неочевидное — фиксируем здесь)

- **`.mm2` = бинарный Delphi DFM/TPF0-стрим**, сериализующий объекты класса `TMudRoom2`,
  кодировка **CP1251**. Один файл ≈ одна зона/область. В эталонном корпусе ~90 файлов →
  **8034 комнаты в 81 зоне** (числа ниже — из этого корпуса; реальные архивы отличаются).
- **Нет vnum'ов.** MUD не отдаёт клиенту числовые id комнат. Идентификация — **фингерпринт
  текста (`hint`+`desc`)** + dead-reckoning (учёт того, каким выходом ты пошёл).
- **`Tag` (int8) НЕ уникален** — повторяется в пределах зоны (до ~79 раз), ~732 null в корпусе.
  Это локальная метка, **не ключ**.
- **`(zone, x, y, l)` уникально** (8034/8034 в корпусе) — наш первичный ключ комнаты. `l` = слой (этаж).
- **Двойные координаты:** `X,Y,L` — визуальная сетка (где рисуем), `DX,DY,DL` — логическая
  «домашняя» позиция. Различаются для **смещённых комнат** (длинные коридоры/наложения): формат
  отделяет нарисованную позицию от логической.
- **Осевая транспозиция:** в игре X = ось С/Ю, Y = ось З/В. На экране: экранный X = координата `y`,
  экранный Y = координата `x`. Дельты шага: **N→x−1, S→x+1, E→y+1, W→y−1, U→l+1, D→l−1**.
- **`ChN..ChD`** — 6 булевых флагов **реальной связности** (N,S,E,W,U,D). **Авторитетнее строки
  `exits`**: `exits` может рекламировать выход, которого нет на карте (`ChN=False`). Использовать
  для рёбер рендера и для neighbor-поиска трекера.
- **Строка выходов живого MUD содержит дверные маркеры:** напр. `(N) E S W` — скобки = дверь;
  и может перечислять немапнутые выходы. Перед фингерпринтом дверные маркеры/скобки **срезать**.
- **`Pipe=True`** — узкие коридоры: **нет Tag, пустой Hint** → нечего фингерпринтить (и, возможно,
  нет стандартного блока комнаты в выводе).
- **`is_dt`** (death trap) выводится из `BColor ∈ {clRed, clMaroon}` (или их int-эквивалентов).
- **`imageindex`** — индекс глифа спецкомнаты (банк/кузница/храм/…); в корпусе 423 спецкомнаты;
  рендерер маппит в эмодзи.
- **`automaps`** — межзонные швы (в корпусе 241 шов): список `[targetZone, targetTag]`, склеивающий
  зоны в единый мир.
- **`bcolor`** — цвет фона комнаты (нашим рендерером не используется, но храним).
- **`desc`** есть не всегда (brief/тёмные комнаты; в корпусе 7358/8034). Русская проза, `\r\n`,
  5 ведущих пробелов — **чистить** перед фингерпринтом.
- **Нормализация имён зон в NFC** обязательна на импорте: часть источников хранит имена в NFD
  (например, экспортированные на macOS) — иначе швы с «й/ё» не резолвятся.

> **Про числа выше:** структурные факты (нет vnum'ов; Tag не уникален; `(zone,x,y,l)` уникален;
> двойные координаты; транспозиция; дельты; `ChN..ChD` авторитетнее exits; дверные маркеры; pipe
> без Tag/Hint; is_dt из BColor; automaps-швы; NFC) — **инварианты формата, на них можно опираться**.
> Мелкие количества (≈79 повторов Tag, ~732 null, 423 спецкомнаты, 241 шов, 7358/8034 с desc) —
> **иллюстрация из эталонного корпуса, НЕ инварианты**: в тестах не ассертить, реальные архивы иные.

## Goals

1. Залив **архива карт** в клиент (админка/Settings): распаковка, выцепление всех `.mm2`,
   склейка в единый мир и создание сущности **«набор карт» (map set)**. Новый архив → новый набор.
2. **Выбор набора карт в сессии** (в разных сессиях — разные миры).
3. Read-only авто-карта как **буфер `~map`**, монтируемый в любой сплит.
4. Следование за игроком без GMCP: авто-ресинк по фингерпринту + backend-трекер dead-reckoning
   со сверкой на каждом шаге и честным состоянием «потерян».
5. Схема заранее держит **картинку на комнату** («вайб»), генерируемую внешним LLM-пайплайном.
6. Не трогать критический путь вывода MUD (Performance-First, AGENTS.md #3).

## Non-goals (в этой итерации)

- GMCP/MSDP-хендлер (структурно — место в `telnet.go`, но за рамками).
- **Генерация** картинок комнат в клиенте — они приходят готовыми из внешнего пайплайна;
  клиент их только хранит и показывает.
- **Трекинг мобов/сущностей — не сюда.** Карта статична по комнатам/топологии; `mud_where`/
  `mud_look_map`/`mud_path` не находят **бродячего** моба (напр. мигрирующий по клеткам NPC).
  `mud_path` доведёт до комнаты, дальше — обычный геймплей через `mud_send_command`.

> **В скоупе (в отличие от GMCP):** MCP-поверхность картографии (§7) — часть фичи, не отложена.
> Read-only тулы (`mud_map_sets`,`mud_map_zone`) — фаза 1; тулы поверх трекера — фаза 3.

## Ключевые архитектурные решения

- **Наборы карт — самостоятельные глобальные сущности, сессия ссылается на набор.** Не
  по-профильная раскладка (осознанное исключение из per-profile паттерна AGENTS #1: карты мира
  от профиля не зависят). Слоистость соблюдаем на уровне «сессия → active_map_set_id».
- **Трекер живёт в Go (session/backend), не в UI.** Держит **in-memory индекс комнат и
  фингерпринтов активного набора карт сессии** (авто-ресинк ищет по всему набору, а map-буфер
  тянет лишь одну зону). Позиция вещается клиентам через `ServerMsg{Type:"room_position"}`
  (`broadcastMsg`, `go/internal/session/clients.go:213`).
- **Картинки комнат — опциональный слой поверх схемы**, наполняется отдельным импортом; рендерер
  и так умеет тамбнейлы (см. `webmap/` в артефактах).

## Scope

### 1. Storage (миграция `012_mapper.sql`)
Следующий номер после существующих `001–011` в `go/internal/storage/migrations/`.

**`map_sets`** — набор карт (результат заливки одного архива):
`id, name, source_archive, imported_at, zone_count, room_count, seam_count, note`.

**`rooms`** — комнаты, принадлежат набору:
`map_set_id(FK), zone, x, y, l, dx, dy, dl, tag, hint, desc, exits, edirs(json), doors(json),
ch(int bitmask ChN..ChD), imageindex, note, is_dt, pipe, bcolor, automaps(json), fingerprint`.
Индексы: UNIQUE `(map_set_id, zone, x, y, l)`; индекс `(map_set_id, fingerprint)`; индекс
`(map_set_id, zone)`.
- `fingerprint` = хэш нормализованного `hint`+`desc`+отсортированные `exits` (дверные маркеры/скобки
  срезаны), NFC, trim, свёртка пробелов. **Скоуп фингерпринта — набор карт** (резолв в пределах
  активного набора, не глобально).
- `is_dt` из `BColor`; `ch` — 6-битная маска связности.

**`room_images`** — картинка «вайба» на комнату (наполняется внешним пайплайном):
`map_set_id, zone, x, y, l`, `thumb(BLOB, ≤128px)`, `full_path(TEXT — файл под
data/room_images/<map_set_id>/<zone>_<x>_<y>_<l>.<ext>, чтобы не пухла БД)`,
`prompt, model, seed, generated_at`. Всё nullable/опционально. Полноразмерные картинки — файлами,
тамбнейлы — инлайн-блобом. **Композитный FK на комнату** `(map_set_id,zone,x,y,l)` опирается на
объявленный выше UNIQUE (валидно для SQLite); альтернатива — суррогатный `rooms.id` и FK на него
(чище). Каскад: `map_sets → rooms → room_images` **`ON DELETE CASCADE`**.

**`room_annotations`** — краудсорс-**overlay** поверх комнаты (VISION: слой аннотаций), отдельно от
топологии: `map_set_id, zone, x, y, l`, `dt(bool)`, `hazard(text)`, `note(text)`, `battle_log(text)`,
`author, updated_at`. **Работает и на frozen-наборе** (аннотация не трогает рёбра → fork не нужен).
DT-аннотация augment'ит `rooms.is_dt` для `mud_path`/`mud_look_map` (если помечено — считать DT).

**`sessions`**: `ALTER TABLE sessions ADD COLUMN active_map_set_id` (какой набор карт у этой
сессии) — по образцу `011_session_timezone.sql:1`. **Висячая ссылка/NULL** (набор удалён или не
выбран) → трекер и map-буфер деградируют мягко: пустая карта / 🔴, без падений; не предполагать
валидный набор.

### 2. Import pipeline (admin) — **архив** → набор карт
- **Приём архива** в Settings (Svelte): дроп `.zip` (основной путь). *(Факт: эталонный корпус
  приходил как RAR5; поддержку `.rar` вынести в бэклог как расширение — zip покрывает основной кейс.)*
- **Извлечение:** рекурсивно достать все `*.mm2` из архива, прочее игнорировать.
- **Порт DFM/`TMudRoom2` парсера** (эталон — `rmud_locate.py`) в Go (`go/internal/mapimport/`):
  CP1251-декод, TReader value-типы (Int8/Int16/Int32/ShortString/LString/Set/Collection), извлечение
  полей, NFC-нормализация имён зон, разрешение смещённых координат (`x,y,l` vs `dx,dy,dl`),
  парс `automaps`.
- **Склейка:** все зоны из архива → один набор; швы `automaps` резолвятся в межзонные рёбра
  (после NFC-нормализации имён). Сводка импорта (зон/комнат/швов, нерезолвнутые швы, конфликты).
- **Новый набор на каждый архив** (не перезаписываем прошлые): создаётся новый `map_sets`-ряд;
  повторная заливка того же архива — по кнопке «заменить набор» либо новый набор (на усмотрение
  пользователя).
- **Инвалидация (AGENTS #2):** после импорта backend перезагружает индекс трекера (если этот набор
  активен где-то) и шлёт `ServerMsg{Type:"map_sets_changed"}` / `{Type:"rooms_changed", zones}` →
  Settings обновляет список наборов, открытые map-панели рефетчат.
- **Удаление набора (админ-действие):** каскадит `map_sets → rooms → room_images` в БД **И**
  удаляет каталог `data/room_images/<map_set_id>/` (файлы на диске каскадом БД не покрываются).
  Любая сессия с `active_map_set_id` на удалённый набор → колонка в NULL + мягкий fallback (см. §1).
  Без этого повторные заливки копят наборы/комнаты/тамбнейлы/файлы неограниченно.

### 3. Data feed (REST)
- `GET /api/map-sets` — список наборов; `GET /api/map-sets/{id}/zones`; `GET /api/rooms?map_set=<id>&zone=<z>`
  — комнаты зоны в slim-JSON (`z t h x y l e d ch a p i s` + `dx dy dl` + `img`(0|1)).
  Регистрируется под `r.Route("/api")` (`go/internal/web/server.go:156`), зеркалит `/api/.../hotkeys|groups`.
- **Картинки — БЕЗ инлайна в JSON зоны** (иначе зона на сотни комнат = payload в мегабайты,
  рефетч на каждый `rooms_changed` — противоречит Performance-First). В slim-JSON только
  **флаг наличия картинки** (`img:0|1`). Тамбнейл и полную картинку тянем **лениво**:
  `GET /api/rooms/thumb?map_set=&zone=&x=&y=&l=` (по видимости в вьюпорте) и
  `GET /api/rooms/image?...` (по фокусу комнаты).

### 4. Map buffer (pane) — рендер
- Новый **тип содержимого пейна** «map», системное имя буфера **`~map`** (префикс — чтобы не
  столкнуться с пользовательскими trigger-буферами: `TargetBuffer` — свободная строка,
  `triggers.go:43-54`; имя `~map` **резервируем** — reject в валидации триггеров).
- Map-пейн **исключается** из текстовых путей `render.ts`: `renderPaneBuffer`/`appendEntries`,
  поиск по пейну, scrollback-сплит (`ui/src/render.ts:975-1030`, `518-526`) — они ассертят текстовый
  буфер. Пейн держит canvas и свой стейт, не entries. Layout уже персистится в
  `localStorage['pane-layout']`.
- Порт canvas-рендерера из `webmap/` (артефакты) в TS-модуль `ui/src/map/*.ts`: зоны, зум/пан,
  этажи, швы (клик → переход в зону), эмодзи-спецкомнаты, DT красным, поиск по имени, отрисовка
  тамбнейлов в клетках + полной картинки по фокусу комнаты. Read-only. Источник данных — активный
  набор карт сессии.

### 5. Current-room signal + Tracker (backend-движок локализации)
- **Room-event парсер** — **stateful-детектор в line-пайплайне session**, НЕ в trigger-VM:
  `MatchTriggers` (`triggers.go:10`) — per-line и уже горячий путь (`db-async-persistence.md:477`,
  `vm_sum=345ms/90 строк`), а блок комнаты многострочный и требует накопителя. Дешёвые prefix-проверки
  сначала, regex только на кандидатах; **вне DB/critical path** (AGENTS #3). Эмитит
  `ServerMsg{Type:"room", hint,desc,exits}` в трекер (и в UI).
- **Формат блока (грамматика + сэмплы — `room_block_fixtures.md` в бандле, ответ на Open Q#1).**
  Якорь парсера — **`[ Exits: ]`-блок** (серверо-агностично), НЕ статус-промпт. Структура: заголовок
  (имя комнаты) → N строк «ASCII-миникарта + desc-проза» → строка выходов. Грамматика:
  `[ Exits: <TOK> ... ]` | `[ Exits: none ]`, `TOK = DIR | (DIR)`, где `(DIR)` = **дверь в этом
  направлении** (каждая дверь в своих скобках: `(N) (E) S (W) (D)`), `DIR ∈ N S E W U D`. Строка
  обёрнута ANSI-cyan + `\r` — **снять ANSI/CR до матча**. Серверо-специфичный RU-промпт
  (`… Выходы:СВЮЗ>`) — максимум **вспомогательный сигнал «шаг завершён»**, на него не закладываться.
- **Движение — очередь FIFO pending-moves** (спидволки/алиасы/хоткеи шлют несколько направлений до
  первого блока комнаты). Направления детектим **после раскрытия алиасов/VM**, в точке записи в MUD
  (`ApplyEffects`→`sendFn`, `triggers.go:64-97`), по RU/EN-таблице (`с/ю/в/з/вв/вн` + `n/s/e/w/u/d`,
  ср. `rmud_locate.py:203`). Очередь с cap'ом; правила мутации: **head-cancel** при «не можете идти»
  (снять только голову, остаток — продолжение с той же клетки), **full-flush** при mismatch имени.
- **Предсказание клетки** по дельтам (N→x−1,S→x+1,E→y+1,W→y−1,U→l+1,D→l−1). **Через межзонный шов
  `automaps` координаты меняются разрывно** — предсказание и neighbor-поиск включают seam-цели
  (ср. `world_neighbors`, `rmud_locate.py:507-528`), не только intra-zone соседей.
- **Сверка на каждом шаге** (room-event против предсказанной клетки по `hint`+нормализованным `exits`;
  **толеранс к различиям только в дверных маркерах**):
  - совпало → 🟢, снять голову очереди;
  - «Вы не можете идти» → head-cancel, позиция не двигается;
  - не совпало → индикатор на **последнюю известную комнату**, флаг/анимация **«пропал с радара»**
    (🔴), full-flush; ждём легитимного авто-ресинка или ручного ре-якоря.
- **Авто-ресинк (легитимный)** по фингерпринт-индексу активного набора — когда фингерпринт
  резолвится в **ровно одну** `(zone,x,y,l)`: уникальный `hint`, **ИЛИ** неуник.`hint` + уник.`desc`
  + уник.`exits`. Иначе авто-ресинка нет → **живём трекером** (🟡).
- **Крайние случаи:** фолловинг/телепорт (очередь пуста) → поиск среди соседей (включая seam-цели)
  → нашли: это направление; нет → 🔴. **Brief/тёмные** (нет `desc`) → только `hint`+`exits`.
  **Pipe-коридоры** (нет Tag/Hint) → держим **🟡 на dead-reckoning, НЕ флипаем в 🔴** из-за
  отсутствия hint для сверки. DT-клетки помечены `is_dt`.

### 6. UI controls
- Выбор **набора карт** в настройках сессии (`active_map_set_id`).
- Открыть/закрыть буфер `~map` (в сплит).
- **«Я здесь»** — ручной ре-якорь (пикер) для «потерян»/старта сессии.
- Тумблер **«следовать за игроком»** (автоцентровка вкл/выкл).
- Индикатор уверенности: 🟢 привязан / 🟡 только трекер / 🔴 потерян.

### 7. MCP tool surface (картография для агента-игрока)
Позиция/карта сейчас уходят **только** в браузерные UI-клиенты (REST + `ServerMsg`-broadcast).
**MCP-агент не потребляет ни то, ни другое** (`/mcp` — request/response, тулы дёргают `s.store`/
`s.manager` напрямую). Без этого раздела вся картография агенту невидима: он может `mud_send_command "с"`,
но не может спросить «где я», прочитать карту или проложить маршрут — а именно **позиция трекера**
самое ценное здесь. Тулы — тонкие **read-only** обёртки над тем же backend-стейтом, что и для UI.

Каждый тул = одна запись в `mcpListTools()` + один `case` в `mcpCallTool()` (`go/internal/web/mcp.go`),
`session_id` опционален через `mcpResolveSessionID`, ответ — текстовый `content`-блок (с `isError`).

- **`mud_where`** — «где я». Вход `{session_id?}`. Выход: активный набор (id+name) или «(none)»;
  уверенность `green|yellow|red`; **`pending_moves:N`** — сколько шагов ещё в FIFO-очереди (§5) не
  подтверждены room-event'ом; `zone,x,y,l,tag,hint`; флаги `is_dt,pipe`; для yellow/red — причина
  («dead-reckoning, нет резолва» / «lost — mismatch на шаге»). Это стейт трекера (§5) по запросу.
  **`pending_moves` ортогонален уверенности:** 🟡 — про *отсутствие резолва*, а `pending_moves>0` —
  про *недослитую очередь*; можно быть 🟢 с 2 неподтверждёнными шагами. Без этого агент, пославший
  батч `с;з;с` и сразу звонящий `mud_where`, примет промежуточный фикс за финальный. То же поле — в
  `mud_look_map`.
- **`mud_look_map`** (он же `mud_room`) — «опиши текущую комнату и выходы по карте». Вход `{session_id?}`.
  Выход: `hint/desc`, выходы с дверными маркерами, по каждому выходу связность из авторитетной маски
  `ch` (`ChN..ChD`) с пометкой `mapped|unmapped`, `→zone` если шов, `is_dt` цели если известно.
  Отличается от `mud_get_output` (сырой скроллбэк) — это структурный вид карты.
- **`mud_map_zone`** — «карта зоны». Вход `{session_id?, zone?, map_set?}` (дефолт: активный набор,
  текущая зона). Выход: slim-список (`z t h x y l e d ch a p i s`+`dx dy dl`+`img`) — тот же формат,
  что `GET /api/rooms` (§3), плюс опц. ASCII-сетка. **Пагинация/бонд** — зона может быть на сотни комнат.
- **`mud_map_sets`** — «наборы карт / какой активен». Вход `{session_id?}`. Выход: по каждому набору
  `id,name,zone_count,room_count`, помечен `active_map_set_id`; NULL/висячий → «(no active set)».
- **`mud_path`** — «маршрут отсюда (или от A) до комнаты/зоны X». Вход
  `{session_id?, to_zone? | to:{zone,x,y,l}? | to_hint?, from?}`. Выход: упорядоченный список
  направлений (`с ю в з вв вн`) для скармливания в `mud_send_command`, число хопов, флаги пересечения
  шва и известных `is_dt` на пути (**на DT — refuse/warn**). Экспонирует BFS (`world_neighbors`,
  `rmud_locate.py`), который иначе портируется только для neighbor-поиска трекера. Без него агент видит
  карту, но не может по ней ходить. **Самый ценный тул после `mud_where`.**
- **`mud_anchor_here`** — MCP-эквивалент кнопки «Я здесь» (§6). Вход `{session_id?, zone,x,y,l}` или
  `{fingerprint_pick}`. Нужен, потому что агент попадёт в 🔴 (телепорт/фолловинг) без браузера, чтобы
  ткнуть пикер, и застрянет навсегда. Выход: новые уверенность+позиция.

**CRUD-тулы (фаза 5, write — см. §8):** для агента, который разведывает локацию и **дорисовывает
карту сам**. Форма — как у существующего write-тула `mud_set_variable` (в `mcp.go` allow-list-гейта
сегодня НЕТ; «защита» = запись через единый write-path §8 + `isError` на frozen-наборе, а не
enforcement-механизм). Все пишут по editable-набору, в общий undo-журнал набора. Если нужен явный
write-enable тумблер для MCP-записи — это **новый механизм, спроектировать отдельно**.
- **`mud_room_upsert`** — создать/обновить комнату. Вход: полный набор полей
  `{map_set?, zone,x,y,l, hint?,desc?,exits?,edirs?,ch?,is_dt?,pipe?,imageindex?,note?,dx?,dy?,dl?}`;
  частичный upsert по ключу `(map_set,zone,x,y,l)`. Спецификация полей для агента — раздел «Факты».
- **`mud_room_link`** / **`mud_room_unlink`** — создать/убрать выход-ребро между клетками (или к шву).
- **`mud_room_delete`** — удалить комнату.
- **`mud_room_annotate`** — **лёгкая аннотация** без перерисовки топологии:
  `{map_set?, zone,x,y,l, dt?, hazard?, note?, battle_log?}`. Пишет в overlay `room_annotations`
  (§1), **НЕ требует fork даже на frozen-наборе** (аннотация ≠ правка рёбер; краудсорс-слой из
  VISION: где DT/подвох/лог боя). Отдельно и по правам: пометить «тут меня убило» безопаснее, чем
  менять связность. Топология-CRUD (`upsert/link/unlink/delete`) — по editable-набору (frozen → fork).
- Топология-CRUD на imported (frozen) наборе → `isError` «set is frozen, fork to editable first» (§8).

**Read-only граница (до фазы 5):** до write-фазы тулы отвечают только где/что/как-дойти; всё
перемещение — через `mud_send_command` (движение течёт через `ApplyEffects→sendFn`, где очередь
pending-moves трекера его наблюдает, §5). Write-исключения того же вида, что `mud_set_variable`
(без enforcement-гейта, см. выше): `mud_set_active_map_set` и `mud_anchor_here`. С фазы 5 —
CRUD-тулы выше.

### 8. Write-режим (фаза 5, поверх трекера): autorecord + undo + CRUD
До этого — строго read-only. Write вводится **единым путём** для UI, MCP и ручных правок.
- **Единый write-path (топология).** UI-autorecord, MCP-CRUD и ручные правки **топологии** идут через
  **одну** backend-функцию записи комнаты/ребра: валидация → запись в active editable-набор →
  `rooms_changed` broadcast → запись в undo-журнал. Правило «никаких параллельных путей» **скоупится к
  топологии** (иначе autorecord и агент рассинхронят набор и индекс трекера).
- **Аннотации — намеренно отдельный overlay-путь, ВНЕ топологического write-path** (это не нарушение
  правила выше: overlay не трогает ни топологию, ни индекс трекера, поэтому десинка, от которого правило
  защищает, тут нет). Семантика: **edit-in-place по `updated_at`, НЕ в undo-стеке** (undo-журнал скоупится
  к editable-набору, а аннотации живут и на frozen-наборах без такого журнала → аннотации не откатываются
  через Undo, только перезаписываются). Запись аннотации шлёт **`annotations_changed`** (лёгкий
  broadcast) → открытые map-панели и клиенты перерисовывают клетку; для DT-аннотации это обязательно —
  она меняет и рендер, и refuse-поведение `mud_path`.
- **Writable vs imported набор.** Импортированный из архива набор — источник правды пользователя;
  чтобы автозапись/CRUD его не портили молча, у набора флаг `editable`. Запись в imported (frozen)
  → предложить **fork в editable-копию** (copy-on-write) либо явно разрешить. Autorecord/CRUD
  работают только по editable-набору.
- **Autorecord (UI-тумблер).** При включённом флаге и живом трекере (фаза 3): каждый room-event для
  **неизвестной** клетки создаёт комнату из распарсенных полей (`hint,desc,exits→edirs`, позиция из
  dead-reckoning), шаг — ребро. Порт логики «мапится, пока ходишь». Базовые гварды: на **🔴 пауза**
  (не писать от потерянной позиции); не перезаписывать существующую клетку.
  **Гэп 🟡 на целине:** на неисследованной территории каждая клетка «неизвестна → создать», поэтому
  дрифт dead-reckoning не ловится mismatch'ем (не с чем конфликтовать) — комнаты сажаются по
  смещённым координатам, «выглядя» уверенно, а «не перезаписывать» потом мешает их исправить.
  **Политика (обязательна):** (i) autorecord только из 🟢-заякоренной клетки **или** в пределах N
  шагов от последнего 🟢; **И** (ii) комнаты, записанные при 🟡, помечаются `provisional/unverified`,
  а при последующем легитимном авто-ресинке — **реконсиляция**: provisional-комнаты переносятся/
  сливаются к истинной позиции (не блокируются правилом «не перезаписывать»). Ребро при 🟡 —
  «предполагаемое», не «подтверждённое».
- **Undo (кнопка).** Ограниченный стек write-операций (create room / create edge / update field /
  delete). **Журнал привязан к editable-набору (`map_set_id`), НЕ к сессии** — наборы глобальны и
  шарятся между сессиями (§«Ключевые решения»), иначе undo в сессии A молча откатит записи B по общим
  строкам. Записи через единый write-path **сериализуются per-set**; undo работает по стеку этого
  набора; операции из UI, autorecord и MCP — в один журнал набора. **Персистентность зафиксировать
  явно:** допустимо in-memory (теряется на рестарте), но MCP-записи бывают без подключённого браузера
  — реализатору нужно знать выбор (in-memory vs durable).
- **Перетягивание/релейаут комнат — НЕ здесь, отдельная фаза 6** (ручной релейаут смещённых комнат:
  механизм `X,Y,L` vs `DX,DY,DL`, коллизии, перекладка коридоров — «космос»).

**Контракт свежести/потери:** каждый тул с позицией (`mud_where`,`mud_look_map`,`mud_path`) возвращает
enum уверенности инлайн, чтобы агент не действовал по устаревшему фиксу. На 🔴: `mud_where` отдаёт
последнюю позицию с явным `stale/lost`; `mud_path` **отказывает** (`isError` / «cannot path: position
lost, re-anchor with mud_anchor_here»), а не строит от ложного старта; pipe-коридоры остаются 🟡 (не 🔴).
Швы и `is_dt` — как флаги в `mud_look_map`/`mud_path`, не молча.

## Phasing

1. **Storage + Import(архив→набор) + REST + read-only map buffer + выбор набора в сессии.**
   Браузабельная карта мира без позиции — ценность сразу. **MCP (read-only, данные уже есть):**
   `mud_map_sets`, `mud_map_zone`.
2. **Room-event парсер** (stateful, session-pipeline) → сигнал `room`.
3. **Backend tracker state-machine** (индекс набора, авто-ресинк + FIFO dead-reckoning + сверка +
   seams + «потерян») + `room_position` вещание + UI-контролы. **MCP (обёртки над трекером/BFS):**
   `mud_where`, `mud_look_map`, `mud_path`, `mud_anchor_here` (+ гейтед `mud_set_active_map_set`).
4. **Слой картинок комнат** (`room_images`): импорт бандла картинок из внешнего пайплайна +
   отрисовка тамбнейлов/полных в буфере.
5. **Write (§8):** editable-набор (fork imported) + единый write-path + undo + UI-autorecord +
   **MCP CRUD** (`mud_room_upsert/link/unlink/delete`) + **аннотации** (`room_annotations` overlay +
   `mud_room_annotate`, работают на frozen-наборе). Поверх трекера (фаза 3).
6. **Перетягивание/релейаут комнат** (смещённые комнаты, `DX/DY/DL`, коллизии) — отдельный этап.

- **Парсер `.mm2`:** синтетические фикстуры (руками собранные TPF0-стримы: Int8/Int16/ShortString/

- **Импорт архива:** zip с несколькими `.mm2` → один набор; вложенные папки; мусорные файлы игнорятся;
  повторная заливка → новый набор.
- **Fingerprint/резолв:** уникальный hint; неуник.hint+уник.desc; полностью неоднозначный → «нет
  авто-ресинка»; exits с дверными маркерами → тот же фингерпринт.
- **Трекер:** golden из пар «команда→комната» (`room_block_fixtures.md`): успешный шаг; «не можете
  идти» (head-cancel); спидволк (несколько pending); **фолловинг без исходящей команды** (очередь
  пуста → поиск среди соседей); телепорт → 🔴 → ре-якорь; переход через шов; pipe-коридор (остаётся
  🟡). Парсер выходов: двери `(N)`, `none`, U/D, ANSI-снятие. Дельты — табличный тест.
- **Аннотации:** `mud_room_annotate` пишет overlay на **frozen**-наборе без fork; **не попадает в
  undo-стек** (edit-in-place, `updated_at`); шлёт `annotations_changed`; DT-аннотация меняет рендер и
  вывод `mud_path`/`mud_look_map` (клетка считается DT).
- **MCP-тулы (read):** `mud_where`/`mud_path`/`mud_look_map` возвращают enum уверенности инлайн; на 🔴
  `mud_path` отдаёт `isError` (не строит от ложного старта); `mud_map_zone` соблюдает бонд/пагинацию;
  `mud_anchor_here` восстанавливает из «потерян».
- **Write (§8):** единый write-path (валидация→запись→broadcast→undo); undo откатывает каждый тип
  операции; **журнал per-set** (undo в сессии A не трогает записи B по общему набору); imported набор
  → запись требует fork в editable; MCP CRUD (upsert частичный/полный, link/unlink, delete) и
  UI-правки — в **один** журнал набора.
- **Autorecord:** создание неизвестной клетки; ребро на шаг; **пауза на 🔴**; не перезапись
  существующей; **🟡-на-целине** → комната `provisional`; последующий 🟢-ресинк → **реконсиляция**
  (provisional перенесены/слиты к истинной позиции, не заблокированы «не перезаписывать»);
  autorecord вне окна N-шагов-от-🟢 не пишет.
- **room_images:** импорт бандла, привязка к `(map_set_id,zone,x,y,l)`, отрисовка при наличии,
  fallback на глиф при отсутствии.
