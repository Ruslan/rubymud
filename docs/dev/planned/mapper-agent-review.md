# Ревью `mapper.md` от агента-потребителя MCP

Автор ревью — LLM-агент, который **реально играет на этом сервере через MCP** (`mud_send_command`):
летал в Палантас ~150 шагов вслепую, свипает 8-клеточный склад руками, использует
оффлайн-парсер `rmud_locate.py` **вне** игры. То есть я — целевой потребитель тулов §7.
Ниже: вердикт, что работает, и 3 пункта на шлифовку (не переделку).

## Вердикт
Дизайн зрелый. Фазировка правильная (read-only ценность сразу, write изолирован в фазу 5),
Performance-First уважён, каверзные места не заметены под ковёр: frozen→fork, per-set undo-журнал,
🟡-provisional/реконсиляция на целине, refuse-on-lost в `mud_path`. Как агент я бы жил на
`mud_where` + `mud_path` + `mud_look_map` с первого дня.

## Что решено правильно (не трогать)
- **§7 — позиция уходит агенту.** Это ядро. Сейчас картография агенту невидима; `mud_where`/
  `mud_path` закрывают ровно эту дыру. Самое ценное во всей фиче.
- **Enum уверенности 🟢🟡🔴 инлайн в каждом позиционном туле.** Без него агент действует по
  устаревшему фиксу, а цена ошибки = шаг в DT. `mud_path` **refuse на 🔴** — то самое поведение.
- **Ранжирование `mud_path` как «самого ценного после `mud_where`».** Из опыта — да. Дамп зоны
  на сотни комнат в контекст агенту дорог и почти бесполезен; нужно «дай направления».
- **§8 provisional + реконсиляция на 🟡-целине.** Пойман тонкий баг: dead-reckoning-дрифт на
  неисследованной территории нечем поймать mismatch'ем. Названо явно — обычно забывают.

## Пункты на шлифовку

### 1. Семантика «шаг осел / не осел» между `mud_send_command` и `mud_where`  ⟵ важнее всего
Движение течёт через async FIFO pending-moves (§5). Если агент шлёт батч `с;з;с` и сразу зовёт
`mud_where` — room-event последнего шага уже распарсен или ещё в полёте? **Нужен явный
`pending_moves: N` в ответе `mud_where`** (и, возможно, в `mud_look_map`). Иначе агент примет
промежуточный фикс за финальный. 🟡 «dead-reckoning» это НЕ покрывает — 🟡 про *отсутствие
резолва*, а не про *недослитую очередь*. Это ортогональные состояния: можно быть 🟢-уверенным,
но с 2 ещё не подтверждёнными шагами в очереди.

### 2. Лёгкий тул аннотаций, а не только поле `note` в `mud_room_upsert`
По VISION проекта моут — слой аннотаций (где DT, где подвох, лог боя). Агент хочет флагнуть
«тут меня убило» / «злой моб» **не перерисовывая комнату** целиком. Просится
`mud_room_annotate {ключ, dt?, hazard?, note, battle_log?}` поверх единого write-path (§8).
Сейчас, чтобы пометить DT, надо гнать полный `mud_room_upsert` — трение на самой ценной
(краудсорс-) операции. Аннотация ≠ правка топологии; разъединить их стоит и по правам
(аннотировать frozen-набор безопаснее, чем менять его рёбра).

### 3. Явная граница «мобы vs комнаты»
Карта статична по комнатам. Главная рутинная боль агента на этом сервере — свип **бродячего**
моба (кладовщик мигрирует по 8-клеточному дереву). `mud_path` доведёт до комнаты, но не найдёт
движущийся моб. Это ок и вне скоупа — но одной строкой в Non-goals «трекинг мобов — не сюда»,
чтобы агент не ждал от `mud_where`/`mud_look_map` того, чего там нет.

## Артефакт для Open Question #1 (уже готов)
«Точный формат блока комнаты» вытащен из живого лога (`log/game.log`, 7219
Exits-блоков) → **`mapper-design/reference/room_block_fixtures.md`** (в бандле артефактов рядом
с FINDINGS.md). Там: грамматика `[ Exits: ]` (per-direction скобки для дверей `(N) (E) S (W) (D)`,
`none`, U/D, ANSI-cyan-обёртка), структура блока, глифы карты (`[*] [^] [v]`), 3 полных сэмпла
(двери+вниз / тупик-brief / лестница-U) и **пары «команда→комната»** из MUX-лога для golden-теста
FIFO-трекера (§5), включая фолловинг без исходящей команды.

Ключ: `Выходы:СВЮЗ>` — **серверо-специфичное RU-зеркало** в статус-промпте RMUD; парсер маппера
(для всех серверов) якорится на `[ Exits: ]`-блок, промпт максимум как вспомогательный сигнал
«шаг завершён». Не закладываться на промпт.
